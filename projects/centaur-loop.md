# Centaur Loop

Open-source workbench for human-governed AI feedback loops. Created by finewood2008 (Chinese team). 35⭐, created 2026-05-08.

## What It Solves

Most agent systems optimize *before* output: prompting, tool use, scheduling. Centaur Loop focuses on *after* output: was it approved? Published? Did it work? What should the agent remember next time?

The core concept: **agents execute, humans govern checkpoints, feedback becomes memory for the next cycle.**

## Architecture

**State machine with explicit human gates:**

```
planning → awaiting_plan_review → generating → awaiting_review 
→ awaiting_publish → awaiting_feedback → reviewing_auto 
→ awaiting_memory → cycle_complete
```

Key components:
- `loopEngine.ts` — Core state machine with `advanceLoop()` that switches on `cycle.stage`
- `loopPlanner.ts` — Turns goals + memory + business context into structured plans
- `loopExecutor.ts` — Generates reviewable drafts per task
- `loopReviewer.ts` — LLM-driven retrospective: extracts memory candidates + next-cycle suggestions
- `loopStore.ts` — Zustand store for loop configs, cycles, checkpoints
- `adapters/memory.ts` — localStorage-based memory (search + store)

**Human gate pattern:**
- Each gate has: stage, timeout action (remind/skip/pause), reminder schedule, notification channels
- Checkpoint types: plan_review, draft_review, publish, feedback, confirm_memory
- Notification channels: spirit_bubble, badge, home_card, chat_followup

**Memory model:**
- Four categories: preference, fact, lesson, correction
- Memory candidates generated by LLM reviewer, then human-confirmed before storage
- Confirmed memories are passed as context to the next cycle's planner
- Simple substring search (no embeddings yet)

## Assessment

### Strengths
1. **Human gates as first-class concept** — not an afterthought. The entire state machine is designed around them.
2. **Memory candidate confirmation** — AI proposes what to remember, human decides. This mirrors our `beliefs-candidates.md` pipeline but with a UI.
3. **Feedback → review → memory → next cycle** — The full loop is explicit and inspectable, not implicit.
4. **Chinese team, bilingual** — Good positioning for both markets.

### Weaknesses
1. **Zero tests** — No test files whatsoever. For a state machine product, this is a significant gap.
2. **localStorage only** — Memory adapter is browser localStorage. No persistence beyond tab.
3. **Content marketing focused** — The flagship demo is "content growth loop" (SEO articles). Not general-purpose yet.
4. **0 issues, 0 PRs** — No community engagement yet. Solo project feel.
5. **Memory search is substring-only** — `content.toLowerCase().includes(query)`. No semantic retrieval.

### Design Decisions Worth Noting
- **Reminder escalation**: Gates have `maxReminders` and `remindAfterMinutes`. If a human doesn't respond, the system can skip or pause. This is more structured than our ad-hoc "blocked on Luna" notes.
- **Cycle numbering + cross-cycle memory**: Each cycle knows its number and can access previous cycle's `nextSuggestion`. This creates continuity across runs.
- **MemoryCategory taxonomy** (preference/fact/lesson/correction) maps cleanly to our wiki structure: preference → beliefs-candidates, fact → wiki/cards, lesson → wiki/projects, correction → beliefs-candidates with "correction" tag.

## Relevance to Our Direction

### Direct parallels
| Centaur Loop | Our System |
|---|---|
| MemoryCandidate (pending→confirmed→rejected) | beliefs-candidates.md (triple verification gate) |
| Human gates with reminders | "Blocker 必须 @ Luna" in AGENTS.md |
| Cycle review → memory extraction | FlowForge reflect nodes |
| nextSuggestion across cycles | Daily handoff notes |

### What we could learn
1. **Structured reminder escalation** — Our "OpenClaw upgrade day 10" keeps getting flagged but never acted on. Centaur Loop's gate pattern with `timeoutAction: 'skip'` would auto-resolve stale blockers instead of repeating the same flag daily.
2. **Memory candidate UI** — Our beliefs-candidates pipeline is file-based and invisible. A confirmation step where Luna can approve/reject memory candidates would reduce noise in beliefs-candidates.md.
3. **Feedback as explicit stage** — We skip the "did it work?" step. After pushing a PR, we track merge status but not *impact*. Centaur Loop makes feedback collection a mandatory stage.

### Where we're ahead
- Our memory system (MEMORY.md + wiki/ + beliefs-candidates) is more mature than their localStorage-based memory
- Our FlowForge workflows are more flexible than their hardcoded stage machine
- We have real-world production experience (30+ PRs, multi-repo, multi-platform)

## Verdict

**Interesting conceptual framework, weak implementation.** The human-governance-first design philosophy is valuable — especially the gate pattern with escalation. But 35⭐, zero tests, zero community, localStorage-only memory means this is a design document more than a usable tool.

**Track?** Optional. Check back in 2 weeks (05-26) to see if it gains traction or stays dormant.

**Actionable insight:** The structured gate escalation pattern (remind → skip → pause) should inform how we handle stale blockers. Currently our daily review just re-flags the same issues. A timeout mechanism would be more effective.
